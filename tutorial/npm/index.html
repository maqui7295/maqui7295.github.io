Working with NPM

Introduction

NPM or node package manager is one of tool for managing dependencies of javascript or Nodejs projects.

Typical workflow of an npm project

git clone repo
cd directory
npm i
npm start
npm test

Getting help

npm -h
npm install -h
npm help <command> # opens browser to the official help page e.g. npm help install
npm help-search <list of words>  e.g npm help-search remove
npm api-help prune
npm info pkg

Setting default configuration

npm set init-author-name "name of author"
npm set init-license "name of author"
npm get init-author-name
# delete a config
npm config delete init-author-name
ls ~/.npmrc # see all options

Starting a new project

npm init # follow the prompt
npm init -y  # same effect as above but faster

installing packages

npm i <pkg>  # same as npm i <pkg@latest>
npm i <pkg> -S or --save
npm i <pkg> -D or --save-dev
npm i -g <globalpackage>

Installing a package from git

npm i <git_repo_url>
npm i <url> # from any other url

Installing from gist

npm i gist:hash_from_url --save (gist url must have a pkg.json file>

Installing from a folder

npm i path_to_folder

Listing installed packages

npm list  
npm ls
npm ll
npm la
npm list --depth 1
npm list --global
npm list --global depth 0
npm list --depth 0 --long true
npm list --depth 0 --json true
npm list --depth 0 --parseable true
npm list --depth 0 --dev true
npm list --depth 0 --prod true

Installing specific versions

NPM packages are semantically versioned i.e. major.minor.patch e.g. pkg1.8.3. Here one(1) is the major. eight(8) is the minor while three(3) is the patch.

  • A revision or patch consists of a bug fix or performance improvement to the package
  • A Minor change usually consists of new features but doesn’t break the existing functionality of the package
  • A Major version is released when package updates or new features introduced break existing functionalities e.g. a change in API or definition of functions

Examples:

npm i <pkg@1.1.1>, npm i <pkg@1.1.x>, npm i <pkg@1.1>, npm i <pkg@1.x>, npm i <pkg@1>

npm i <pkg@">=1.1.0 < 1.4.0 || 1.2.x">
npm i <pkg@"<4">
npm i <pkg@"3">

npm i <pkg@"^1.7.4">

npm i --save # npm automatically updates to latest version

npm i --save --save-exact <pkg@1.8.2> installs exact version:  "1.8.2" no caret symbol 

caret(^) --- get the latest major release e.g. "^1.2.1"
tilde(~) --- get the latest version of the minor release e.g. "~1.2.1"

asterisks(*) or "*" e.g. "*" or "x" --- get the latest package.

Updating versions

npm update   # update all local pkgs
npm update --dev  # update devDependencies
npm update --prod  # update dependencies
npm update <pkg> # update a single pkg
npm update -g  # be careful about this especially if you have much pkgs installed globally
npm update -g <globalpkg> # update a single global package

npm i npm@latest -g # with admin priviledges

Removing packages

npm uninstall <pkg>
npm uninstall <pkg> --save # both uninstalls and remove pkg from pkg.json file
npm rm <pkg>
npm r <pkg>
npm un <pkg>
npm rm <pkg>
# add the -g flag to the above command to remove a global pkg

npm prune # remove an extraneous pkg
npm prune pkg  # equivalent to npm rm pkg
npm prune --production  # remove all devDependencies

NPM registry

registry.npmjs.org/<pkgname>  # get info on pkgname in json format

npm.im/pkg   # takes you directly to the package url on npmjs.com

npm search pkg

npm repo pkg  # the repo page of the pkg

Publishing an NPM package

STEPS:

  1. create a user account on npmjs.com
  2. npm adduser # enter your username and password

  3. Prepare the project for publishing

    a. git init
b  git add orin <url_git_repo?
c. npm init
d. git add .
e. git commit -m "message"
f. git push

  4. Run npm publish. Check or verify the status of the package. npm info pkg or npm repo pkg or npm.im.pkg

  5. Tag the package: git tag <tag_id>

  6. git push --tags # this ensures that the npm version matches the source on github>

Publishing an update

The steps are:

  1. Make changes then commit them as you would in a git project.
  2. Update the version either manually or via npm version <patch/minor/major>

  3. If npm version <patch/minor/major> command was used, npm automatically tags and commits the changes, then

    a. git push --tags <tag-id>
b. git push
c. npm publish

Releasing a beta version

  1. Manually change the version in the package.json “1.1.0-beta.0” # note the beta.n word
  2. Commit the changes
  3. Tag it with git e.g. git tag 1.1.0-beta.0
  4. git push
  5. git push --tags
  6. npm publish --tag beta
Run npm i pkg@beta to install the beta version
